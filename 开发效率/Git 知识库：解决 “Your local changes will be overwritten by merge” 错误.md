

# Git 知识库：解决 “Your local changes will be overwritten by merge” 错误

## 1. 错误背景
当你尝试执行 `git pull` 或 `git merge` 时，如果 Git 发现远程分支的代码与你本地**尚未提交（Uncommitted）**的修改在同一个文件中有冲突，就会触发此错误。Git 为了防止你的本地代码被意外覆盖，会中断合并操作。

---

## 2. 解决方案
根据你对本地修改的处理意愿，可以选择以下三种方案：

### 方案 A：暂时存放修改（最推荐）
**适用场景**：本地代码写了一半，还没到提交的时候，但需要先更新远程代码。

1. **暂存本地修改**（将改动存入堆栈）：
   ```bash
   git stash
   ```
2. **拉取远程代码**：
   ```bash
   git pull
   ```
3. **恢复本地修改**（将堆栈中的改动合并回代码）：
   ```bash
   git stash pop
   ```
   *注意：如果在执行 `pop` 时出现冲突，请根据提示手动修改冲突文件。*

---

### 方案 B：正式提交修改
**适用场景**：本地改动已经完成，希望将其作为一次正式的历史记录保存。

1. **添加并提交**：
   ```bash
   git add .
   git commit -m "保存本地修改以便合并"
   ```
2. **拉取远程代码**：
   ```bash
   git pull
   ```
   *注意：此时 Git 会自动尝试合并。如果文件改动位置不同，会自动生成一个 "Merge commit"；如果改动了同一行，则需要手动解决冲突后再提交。*

---

### 方案 C：放弃本地修改（慎用）
**适用场景**：本地的修改不想要了，或者改乱了，想完全以远程代码为准。
**警告：此操作不可逆，本地未提交的代码将永久丢失！**

*   **方式 1：只针对特定文件**
    ```bash
    git checkout -- <文件名>
    git pull
    ```
*   **方式 2：重置整个工作区（最彻底）**
    ```bash
    git reset --hard HEAD
    git pull
    ```

---

## 3. 决策指南（快速查阅表）

| 你的需求 | 推荐操作 | 核心命令 |
| :--- | :--- | :--- |
| **我想先保留改动，回头再接着写** | **方案 A (Stash)** | `git stash` -> `pull` -> `git stash pop` |
| **我的代码写完了，准备一起推送到分支** | **方案 B (Commit)** | `git commit` -> `pull` |
| **我的本地改动没用了，直接用远程的** | **方案 C (Discard)** | `git reset --hard HEAD` |

---

## 4. 预防建议
*   **勤提交**：养成频繁 `git commit` 的习惯，提交后再 pull 就不会遇到这个阻碍。
*   **先拉后写**：在开始新的一天开发前，先执行 `git pull` 同步最新代码。
*   **分支隔离**：尽量在功能分支（Feature Branch）上开发，避免在主分支直接修改。
