# 企业级 AI Agent 架构实战：基于 Spring Boot 3.4.4 + Java 21 + LangChain4j

随着大语言模型（LLM）进入爆发期，如何构建一个稳定、高性能且易于维护的 AI Agent 成了开发者的核心课题。本文将基于最新的 **Spring Boot 3.4.4** 和 **Java 21**，深度解析在集成 **Redis**、**Milvus** 向量数据库以及 **LangChain4j** 时遇到的坑点与最佳优化实践。

## 1. 技术栈核心选型

*   **JDK**: 21 (利用虚拟线程 Virtual Threads 处理高并发 AI 调用)
*   **Framework**: Spring Boot 3.4.4 (目前最新的稳定版)

## 2. Redis 优化：解决序列化与性能瓶颈

在 Spring Boot 3.x 中，Redis 的配置应遵循“约定大于配置”原则，重点在于自定义序列化器以避免可视化工具查看时的“乱码”。

### 最佳实践：
*   **序列化器**：使用 `Jackson2JsonRedisSerializer` 并注册 `JavaTimeModule`，确保 `LocalDateTime` 等类型能正确解析。
*   **类型保留**：通过 `activateDefaultTyping` 确保 JSON 存入 Redis 后能反序列化回正确的 Java 对象。

```java
@Bean
public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setConnectionFactory(factory);

    ObjectMapper om = new ObjectMapper();
    om.registerModule(new JavaTimeModule()); // 支持 Java 8 时间 API
    om.activateDefaultTyping(LaissezFaireSubTypeValidator.instance, 
                             ObjectMapper.DefaultTyping.NON_FINAL, 
                             JsonTypeInfo.As.PROPERTY);

    Jackson2JsonRedisSerializer<Object> serializer = new Jackson2JsonRedisSerializer<>(om, Object.class);
    template.setKeySerializer(RedisSerializer.string());
    template.setValueSerializer(serializer);
    return template;
}
```

## 3. Milvus 向量库集成：从 V1 到 V2 的跨越

Milvus 2.4+ 推出了 V2 SDK，API 更加函数式且支持 URI 连接。

### 踩坑提醒：Database Not Found
在 Spring Boot 启动时，如果配置文件中指定的 `database-name` 在服务器上不存在，应用会直接抛出 `BeanInstantiationException` 导致启动失败。

### 解决方案：自动检查并初始化数据库
```java
@Bean(destroyMethod = "close")
public MilvusClientV2 milvusClientV2() {
    // 步骤1：先用 default 库连接，检查目标数据库是否存在
    ConnectConfig rootConfig = ConnectConfig.builder().uri(uri).dbName("default").build();
    try (MilvusClientV2 tempClient = new MilvusClientV2(rootConfig)) {
        if (!tempClient.listDatabases().contains(targetDb)) {
            tempClient.createDatabase(CreateDatabaseReq.builder().databaseName(targetDb).build());
        }
    }

    // 步骤2：连接到业务数据库
    return new MilvusClientV2(ConnectConfig.builder()
            .uri(uri).dbName(targetDb).build());
}
```

## 4. LangChain4j 核心冲突：Multiple HTTP Clients 异常

在集成 `langchain4j-open-ai-spring-boot-starter` 1.13.0 时，由于类路径下同时存在 JDK HTTP Client 和 Spring RestClient，系统会报如下错误：
`Conflict: multiple HTTP clients have been found in the classpath`

### 深度修复方案
这是由于 LangChain4j 的 `ServiceLoader` 机制在 Spring 容器启动前就尝试加载 Client。

1.  **强制指定系统属性**：在 `main` 方法第一行指定工厂。
    ```java
    System.setProperty("langchain4j.http.clientBuilderFactory", 
        "dev.langchain4j.http.client.spring.restclient.SpringRestClientBuilderFactory");
    ```
2.  **Maven 排除冲突项**：
    ```xml
    <dependency>
        <groupId>dev.langchain4j</groupId>
        <artifactId>langchain4j-open-ai-spring-boot-starter</artifactId>
        <exclusions>
            <exclusion>
                <groupId>dev.langchain4j</groupId>
                <artifactId>langchain4j-http-client-jdk</artifactId>
            </exclusion>
        </exclusions>
    </dependency>
    ```

## 5. 现代化配置：Java 21 Record + 属性校验

不要再使用散乱的 `@Value`。利用 Java 21 的 `record` 配合 `@ConfigurationProperties`，可以让配置代码更简洁、安全。

```java
@Validated
@ConfigurationProperties(prefix = "openai")
public record OpenAiProperties(
    @NotBlank String apiKey,
    @DefaultValue("gpt-4o") String model,
    @DefaultValue("0.7") Double temperature,
    @DefaultValue("60") Integer timeoutSeconds
) {}
```

## 6. 性能飞跃：开启虚拟线程 (Virtual Threads)

AI Agent 是典型的 I/O 密集型应用（等待 LLM 响应、等待向量库检索）。在 Spring Boot 3.4.4 中，只需一行配置即可开启虚拟线程，极大提升吞吐量。

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

## 7. 总结：迈向生产级的 Agent

构建 AI Agent 不仅仅是调用 API，更是一场关于依赖管理、资源初始化和高性能架构的挑战。

*   **安全性**：API Key 永远通过环境变量注入，不要写死。
*   **鲁棒性**：在配置类中增加数据库/集合的自动检查逻辑。
*   **可观测性**：引入 `opentelemetry` 追踪 Agent 的决策路径。

通过本文的优化方案，你的 AgentX 框架将拥有目前 Java 生态中最前沿的技术底层，从容应对高并发与复杂语义检索的挑战。
