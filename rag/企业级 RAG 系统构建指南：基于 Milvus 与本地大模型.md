## 🏗️ 1. 架构总览：RAG 系统的核心流转

在构建检索增强生成（RAG）系统时，我们需要解决让 AI “懂私有数据”并“严谨回答问题”的痛点。整个系统流水线由三个核心环节构成：

1. **翻译官（Embedding 模型）**：将人类的自然语言文档（如 PDF、Word）转化为机器能理解的“高维向量数组”（即一堆数字）。
2. **记忆库（向量数据库）**：负责接收这些庞大的向量数据，建立索引。当用户提问时，在亿级数据中实现毫秒级的相似度检索。
3. **决策大脑（大语言模型 LLM）**：接收用户的原始问题和数据库检索出的参考背景，进行阅读理解，并生成最终的人类语言回答。

## 🗄️ 2. 核心基础设施：Milvus 向量数据库

**Milvus** 是一款由 Zilliz 发起、LF AI & Data 基金会顶级毕业项目的开源云原生向量数据库。

### 2.1 为什么选择 Milvus？
* **海量规模**：专为千万级、亿级甚至万亿级数据集设计。
* **存算分离架构**：查询节点（Query Node）、数据节点（Data Node）和索引节点（Index Node）完全解耦，可按需独立扩容，资源利用率极高。
* **混合搜索 (Hybrid Search)**：内置 BM25 算法，支持**稠密向量（语义） + 稀疏向量（关键词） + 标量过滤（如价格区间）**的融合检索。

### 2.2 模型 vs 数据库的通俗比喻
如果把构建知识库比作建一个图书馆：
* **Embedding 模型（如 `bge-m3`）是“图书编目员”**：阅读书籍，理解内容，并打上索书号（生成向量）。算完即走，**不负责存储**。
* **Milvus 是“智能书架系统”**：它看不懂中文，但能根据索书号的规则（索引），把海量数据排列好。当用户拿查询向量来时，它能瞬间指出相似数据的位置。

### 2.3 同类产品对比
| 特性 | **Milvus** 🚀 | **Pinecone** | **Weaviate** | **Qdrant** |
| :--- | :--- | :--- | :--- | :--- |
| **开源属性** | ✅ 完全开源 | ❌ 闭源 (SaaS) | ✅ 开源 | ✅ 开源 |
| **核心优势** | 极致性能、十亿级数据处理 | 极易上手 (Serverless) | 混合搜索体验好 | Rust 编写，资源占用低 |
| **部署方式** | K8s, Docker, Cloud, Python Lite | 仅限云端 | Cloud, Docker | Cloud, Docker |

## 🧠 3. 模型层解析：大模型与嵌入模型

在本地化部署（如使用 Ollama）时，我们需要两个不同类型的模型相互配合：

### 3.1 翻译官：Embedding 模型 (如 `bge-small-zh` / `bge-m3`)
* **本质**：文本向量化工具。
* **特点**：体积小巧，不具备对话能力，但极其擅长“判断两个句子的语义相似度”。
* **代表作**：智源研究院出品的 BGE (BAAI General Embedding) 系列，长期霸榜中文检索评测榜单。

### 3.2 决策大脑：LLM (如 `moe-7b` / `qwen:7b`)
* **本质**：负责阅读理解与生成的语言模型。
* **概念科普（MoE）**：MoE (Mixture of Experts) 指**混合专家架构**。它由多个“小专家模型”组成，遇到不同问题时只激活对应的专家（如编程专家、财务专家）。优势是**参数量大但推理速度极快**。
* **选型推荐**：对于政策问答等严谨场景，推荐使用 **Qwen 7B/14B（通义千问）**，其中文逻辑推理能力处于开源顶尖水平。

## 📊 4. Embedding 模型选型指南

建议通过 Ollama 本地部署以下模型（注重隐私与成本控制）：

| 模型名称 | 厂商 | 输出维度 (Dimension) | 适用场景与优势 | Ollama 拉取命令 |
| :--- | :--- | :--- | :--- | :--- |
| **`bge-m3`** | 智源 | **1024** | **🔥 首选全能王**。支持多语言、长文本 (8192 token)、混合检索。 | `ollama pull bge-m3` |
| **`bge-small-zh`** | 智源 | **512** | **资源受限环境**。速度极快，适合无 GPU 或 CPU 环境。 | `ollama pull bge-small` |
| **`jina-embeddings-v3`**| Jina AI| 1024 | **超长文本之王**。适合单次灌入超长法律条文或财报。 | `ollama pull jina/jina-embeddings-v3` |
| **`gte-large`** | 阿里 | 1024 | **通义生态**。如果你大模型用 Qwen，用 GTE 配合效果更好。 | `ollama pull gte-large` |

> *注：若数据允许出公网且不差钱，可直接调用 OpenAI 的 `text-embedding-3-small` (1536维)。*

## ⚙️ 5. 实战配置：Spring AI 核心参数详解

以下是一份标准的 RAG 应用 `application.yml` 配置（以政策问答系统为例），每一行都经过深度调优：

```yaml
ai:
  ollama:
    # Ollama 本地服务地址 (若在 Docker 桥接网络中，可能是 host.docker.internal)
    base-url: http://localhost:11434  
    
    chat:
      model: qwen:7b  # 🧠 决策大脑：通义千问 7B
      options:
        # 🌡️ 创造力控制 (0.0 - 1.0)
        # 0.3 代表高度严谨保守。对于政策文档问答，必须防止模型“胡编乱造”（幻觉），严格依赖检索出的上下文。
        temperature: 0.3 
        
    embedding:
      model: bge-m3   # 🔎 翻译官：智源 BGE-M3 (需在 Ollama 提前 pull)
  
  vectorstore:
    milvus:
      client:
        host: localhost
        port: 19530   # Milvus API 通信端口 (切勿填成管理面板端口)
      collection-name: policy_docs # 🗂️ Milvus 集合名称 (类似数据库表名)
      
      # 📏 向量维度配置
      # 必须与上方 Embedding 模型的输出维度严格一致！(bge-m3 输出维度为 1024)
      embedding-dimension: 1024 
```

### ✅ 部署检查清单 (Checklist)
启动服务前，请务必在终端验证以下状态：
1. **Milvus 服务就绪**：`docker ps | grep milvus`
2. **LLM 模型已加载**：`ollama run qwen:7b` (测试一下是否能对话)
3. **Embedding 模型已加载**：`ollama pull bge-m3` (因体积较大，首次需等待)

## 🚨 6. 避坑指南：致命的“维度陷阱”

在 RAG 系统开发中，新手最容易踩的坑是**更换模型导致的维度崩溃**。

### 💣 故障场景
1. 系统初始配置使用 `bge-m3` (1024维)，成功运行，Milvus 创建了名为 `policy_docs` 的集合。
2. 业务需求变更，将配置文件的模型换成了 `bge-small-zh` (512维) 或接入 OpenAI API (1536维)。
3. 重启程序后，**服务疯狂报错，数据无法入库**。

### 🔍 根本原因
Milvus（以及几乎所有向量数据库）中的 Collection 一旦创建，其**字段维度（Dimension）是永久固定的**。你不能把 512 维的数组强行塞进 1024 维的坑位里。

### 🛠️ 标准解决流程 (三部曲)
如果你必须更换 Embedding 模型，请严格按以下步骤操作：
1. **删库清空**：通过代码或可视化工具（如 Attu）删除 Milvus 中旧的 `policy_docs` 集合。
2. **修改配置**：同步更新 `application.yml` 中的 `embedding-dimension` 参数（如改为 512 或 1536）。
3. **重新跑批**：重启 Java/Python 应用，触发文档的重新切片（Chunking）和向量化入库（Embedding）。
