# 政策文档智库系统：全栈开发实战文档

## 1. 项目概述
本系统旨在建立一个基于 RAG（检索增强生成）技术的政策咨询助手。用户通过上传政策文档（PDF/TXT），系统将其向量化存储至 Milvus 数据库。提问时，AI 结合检索到的政策片段，给出精准、有据可查的回答。

### 技术栈
*   **后端**：Spring Boot 3.2+, Spring AI 2.x (Milvus, OpenAI/DeepSeek 适配器)
*   **AI 模型**：DeepSeek-V3 / R1
*   **向量数据库**：Milvus
*   **前端**：Vue 3 (TypeScript), Vite, Markdown-it
*   **通信协议**：SSE (Server-Sent Events) 流式传输

## 2. 后端核心实现 (Spring AI)

### 2.1 响应式 RAG 服务
使用 `QuestionAnswerAdvisor` 简化 RAG 流程，并集成 **超时保护** 与 **异常兜底**，防止后端故障引发客户端无限重试。

```java
@Service
public class RagService {
    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public RagService(ChatClient.Builder chatClientBuilder, VectorStore vectorStore) {
        this.vectorStore = vectorStore;
        this.chatClient = chatClientBuilder
            .defaultSystem("你是一位政策专家，请基于背景资料回答。使用 Markdown 格式。")
            .build();
    }

    public Flux<String> streamAnswer(String query) {
        return Flux.defer(() -> {
            // 1. 检索阶段 (带超时控制)
            List<Document> docs = vectorStore.similaritySearch(
                SearchRequest.builder().query(query).topK(3).similarityThreshold(0.6).build()
            );

            if (docs.isEmpty()) return Flux.just("🔍 未找到相关政策。");

            String refs = docs.stream()
                .map(d -> (String) d.getMetadata().getOrDefault("filename", "未知"))
                .distinct().collect(Collectors.joining(", "));

            // 2. 生成阶段 (Advisor 模式)
            return chatClient.prompt().user(query)
                .advisors(new QuestionAnswerAdvisor(vectorStore, SearchRequest.defaults()))
                .stream().content()
                .timeout(Duration.ofSeconds(30)) // 防止模型卡死
                .concatWith(Flux.just("\n\n---\n> 📚 **参考文件：** " + refs))
                .onErrorResume(e -> Flux.just("\n\n⚠️ [服务异常] 请稍后再试。")); // 确保返回200
        }).doOnCancel(() -> log.info("客户端主动断开连接"));
    }
}
```

## 3. 前端核心实现 (Vue 3)

### 3.1 极简专业 UI 设计
*   **Markdown 支持**：集成 `markdown-it` 实现标题、列表、加粗渲染。
*   **思考时间追踪**：实时计算从发送请求到 AI 吐出第一个字的耗时。
*   **物理断连**：使用 `AbortController` 实现“停止生成”功能。

### 3.2 流式请求逻辑 (原生 Fetch 版)
弃用带自动重试机制的第三方库，改用原生 Fetch 以获得对“连接积压”和“自动重连”的完全控制。

const sendQuery = async () => {
  abortController.abort(); // 杀掉旧请求
  abortController = new AbortController();
  
  startThinkingTimer(); // 开始计时
  let isFirstChunk = true;

  try {
    const response = await fetch(`${API_BASE}/chat?query=${encodeURIComponent(text)}`, {
      signal: abortController.signal
    });

    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;

      if (isFirstChunk) {

## 4. 关键问题与避坑指南

### Q1: 页面上出现大量的 `data:` 字符？
*   **原因**：Spring WebFlux 默认使用 SSE 协议。每一行有效数据都会带上 `data:` 前缀。
*   **解决**：前端解析时使用 `line.substring(5)` 剥离前缀，或后端强制设置 `MediaType.TEXT_PLAIN_VALUE`（但不推荐，会失去标准 SSE 优势）。

### Q2: 报错后疯狂重连，Milvus 压力爆表？
*   **深度原因**：
    1.  **后端**：返回了 500 错误。
    2.  **前端库**：`fetch-event-source` 默认在收到 500 时会自动重试。
    3.  **浏览器**：同域名并发连接限制（6个），导致新请求进入 `Stalled` 状态，排队报错重连，形成死循环。
*   **终极对策**：
    1.  **后端兜底**：使用 `onErrorResume` 捕获所有异常并返回一段错误文字（HTTP 200）。
    2.  **前端物理熔断**：改用原生 `fetch` 或在 `onerror` 中明确 `throw error` 并调用 `abort()`。

### Q3: 页面样式“乱码”或不理想？
*   **原因**：Tailwind CSS 未正确初始化。
*   **解决**：在组件内使用 `<style scoped>` 编写原生 CSS。针对 Markdown 内容，使用 `:deep(.markdown-body)` 深度选择器来美化生成的 HTML 标签（如 `<ul>`, `<strong>`）。

## 5. 项目亮点总结
1.  **感知增强**：通过“思考时长”标签，量化了 RAG 检索的性能表现。
2.  **鲁棒性**：实现了全链路的异常捕获，确保即便在 AI 服务不稳定时，系统也不会崩溃或自我攻击（无限重连）。
3.  **性能优化**：通过响应式流的 `doOnCancel`，实现了端到端的资源释放，节省了昂贵的 LLM Token 成本。
