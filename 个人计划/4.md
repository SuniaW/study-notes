### 💡 为什么要做这个系统？（被逼出来的极限调优）

目前中小微企业想接入大模型，面临着极其尴尬的三座大山：
1. **语言壁垒**：市面上 90% 的 AI 教程都是 Python，广大的 Java 开发者面临技术栈割裂。
2. **算力刺客**：私有化部署动辄要求 RTX 4090 或 A100 集群，中小企业根本买不起一年几万块的 GPU 服务器。
3. **玩具级 Demo**：网上的教程只教你写个问答脚本，根本不教你如何解决复杂 PDF 解析，更缺乏真实业务中的 Function Calling（工具调用）闭环。

既然没钱买显卡，那能不能用公司退下来的**“穷鬼套餐”（2核4G内存服务器）**跑起完整的企业级 AI 知识库？

答案是：**能，但要把系统压榨到极致。**

#### 狠活一：2C4G 的“极限生存法则”
如何在区区 4GB 的内存里，同时塞下 Spring Boot、Milvus (向量数据库)、Ollama (推理引擎) 还不发生 OOM (内存溢出)？
我查阅了大量底层文档，通过 **Linux 内核 Swap 虚拟内存调优、Swappiness 参数压榨、JVM 精准配额以及 GGUF 模型量化**，硬生生在百元级服务器上跑通了秒级响应的大模型应用！

![alt text](微信图片_20260402094113_35_9.png)

看看这令人极度舒适的内存分配，这就是架构师的强迫症！

#### 狠活二：接入真实商业 API，打造真正的 Agent
客户不会为“聊天机器人”买单，他们只为“能干活的数字员工”买单。
借助 Spring AI 2.x 强大的 `@Tool` 机制，我彻底打通了 DeepSeek 与外部真实世界的连接。

当我问系统：“西安今天天气怎么样？”
它不再是瞎编，而是**自动生成 JSON 指令 -> 触发本地 Java 方法 -> 调用高德真实气象 API -> 获取数据后结合大模型推理，流式输出给用户。**

![alt text](2026-04-02_09-03.png)

仔细看，当我故意在底层代码里返回一个假数据 `-255°C` 时，AI 会凭借其常识推理能力，敏锐地指出：**“这个温度看起来非常低，可能数据有些异常，建议您再次确认。”**
这就是真正意义上的 AI Agent（智能体）与传统死板程序的本质区别！
